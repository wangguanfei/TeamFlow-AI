● 入门

什么是 Claude Code

Claude Code 是 Anthropic 推出的 AI 编程工具，以 CLI（命令行）为核心，深度集成到你的终端、IDE 和工作流中。它不只是代码补全——它是一个能理解整个项目、执行多步任务、调用真实工具的 AI 工程师。

核心能力概览

🛠
技能（Skills）
封装好的专项能力，如代码生成、重构、调试、文档撰写等，通过 /skill-name 调用。
⚡
命令（Commands）
内置斜杠命令，如 /init /compact /cost，控制会话和工作流。
🤖
代理（Agents）
专职 AI 子代理，如 planner、tdd-guide、code-reviewer，可并行执行复杂任务。
📋
规则（Rules）
通过 CLAUDE.md 定义项目级别行为规范，Claude 在整个会话中自动遵守。

与普通 AI 助手的区别

维度普通 AI 助手Claude Code
代码执行 生成代码片段，需要你手动复制运行 直接读写文件、运行命令、查看输出
项目理解 只能看到你粘贴的片段 读取整个代码库，理解架构
工具调用 无法调用真实工具 调用 git、npm、docker、测试框架等
持久上下文 每次对话从零开始 CLAUDE.md + 记忆系统跨会话保持
任务粒度 单次问答 多步骤、多文件、多代理协同
💡
心智模型 把 Claude Code 想象成一个坐在你旁边的高级工程师：你描述需求，它直接动手——读文件、写代码、跑测试、提 PR。你是决策者，它是执行者。

安装与配置

从零开始把 Claude Code 跑起来，只需三步。

1

安装 Claude Code CLI

需要 Node.js 18+。通过 npm 全局安装：

npm install -g @anthropic-ai/claude-code
# 验证安装
claude --version
2

登录并授权

首次运行时会弹出浏览器进行 OAuth 授权，或直接配置 API Key：

claude
# 或者设置环境变量（适合 CI/自动化场景）
export ANTHROPIC_API_KEY=sk-ant-xxxxx
3

在项目中初始化

进入你的项目目录，让 Claude 生成专属配置文件：

cd /your/project
claude
# 在对话中输入：
/init
# Claude 会扫描项目，自动生成 CLAUDE.md

IDE 集成

Claude Code 提供官方 VS Code / JetBrains 插件，安装后可在编辑器内直接唤起，无需切换终端：

IDE安装方式唤起快捷键
VS Code扩展市场搜索 "Claude Code"Cmd/Ctrl + Shift + P → Claude
JetBrains 全系Plugin Marketplace 搜索 "Claude Code"Cmd/Ctrl + Shift + A → Claude
终端（Mac）已安装 CLI 即可直接运行 claude

推荐全局设置

编辑 ~/.claude/settings.json 进行个性化配置：

{
  "theme": "dark",
  "autoUpdates": true,
  "preferredNotifChannel": "terminal",
  // 允许 Claude 直接执行的工具（无需每次确认）
  "allowedTools": ["Bash(git *)", "Bash(npm test)"]
}
⚠️
安全提示 allowedTools 中只添加你信任的命令。危险命令（如 rm -rf）应保持需要手动确认，避免误操作。

运行 /init 生成 CLAUDE.md

/init 是进入新项目的第一条命令。Claude 会扫描目录结构、分析技术栈，自动生成一份项目配置文件：

# 在项目根目录启动 Claude
claude

# 在对话中执行
/init

生成的 CLAUDE.md 包含：

  • 项目技术栈说明（语言、框架、包管理器）
  • 常用命令（构建、测试、启动）
  • 目录结构说明
  • 代码规范和注意事项

手动补充关键信息

自动生成的内容是基础，你应该手动追加团队规范、架构决策和特殊约定：

# CLAUDE.md 追加内容示例

## 架构约定
- 所有 API 请求统一走 src/api/ 目录
- 状态管理使用 Zustand，禁止直接修改 store
- 组件按功能域分目录，不按类型分

## 不可修改的文件
- deploy/ 目录下的配置，改动需 DevOps 审批
- src/lib/crypto.ts，安全关键代码，需专项评审

## 开发环境
- 数据库: PostgreSQL 15，连接串在 .env.local
- 后端端口: 8080，前端端口: 5173
💡
CLAUDE.md 的价值 每次新建会话，Claude 都会首先读取 CLAUDE.md。这是你的"项目记忆"——它让 Claude 在第 100 次对话时仍然了解项目背景，不需要每次重新解释。

子目录 CLAUDE.md

对于大型 monorepo，可以在子模块中放置独立的 CLAUDE.md：

project/
├── CLAUDE.md          # 全局规范
├── frontend/
│   └── CLAUDE.md      # 前端专属规范（Vue/React 约定）
├── backend/
│   └── CLAUDE.md      # 后端专属规范（Java/Go 约定）
└── deploy/
    └── CLAUDE.md      # 部署配置说明

使用 planner 代理

对于复杂功能，先调用 planner 代理输出结构化实现方案，再开始写代码：

# 在对话中描述需求，Claude 会自动激活 planner
我需要为电商系统添加优惠券模块，支持满减、折扣、限量三种类型，
需要与订单系统集成，并提供管理后台 API。请先规划实现方案。

planner 代理会输出：

  • 数据库表结构设计
  • API 接口列表
  • 文件/模块划分
  • 实现顺序（考虑依赖关系）
  • 风险点和注意事项

开启计划模式（Plan Mode）

按 Option+P（Mac）/ Alt+P（Windows）进入计划模式。在此模式下，Claude 只分析和规划，不执行任何写操作，你审批后才开始执行：

ℹ️
计划模式 vs 正常模式 计划模式适合"我知道要做什么但不确定怎么做"的场景，能帮你在 AI 动手前看清全局，避免中途大幅改向浪费 token。

开启扩展思考

对于架构选型、算法设计等复杂决策，按 Option+T 开启扩展思考（Extended Thinking）。Claude 会在内部进行更深入的推理，给出更周全的方案：

# 典型的扩展思考场景
我们要支持百万级用户的实时消息推送，请对比 WebSocket、SSE、
长轮询三种方案的优劣，并给出推荐选型及原因。

拆解复杂功能

如果功能很大，明确告诉 Claude 分阶段实现：

请按以下顺序分三个阶段实现用户权限系统：
阶段一：数据库表结构和基础 CRUD API（不含权限逻辑）
阶段二：RBAC 权限模型和中间件
阶段三：前端权限路由守卫和按钮级别控制

我们先做阶段一，完成后我会说"继续阶段二"。

自定义代理

当内置代理不满足需求时，你可以创建专属代理——它们有自己的系统提示词、工具权限和领域知识。

代理文件结构

代理以 Markdown 文件形式存储在 ~/.claude/agents/ 目录：

~/.claude/agents/
├── planner.md          # 内置代理（可覆盖）
├── code-reviewer.md
└── my-domain-expert.md # 你的自定义代理

创建自定义代理

代理文件由 YAML frontmatter 和 Markdown 系统提示词组成：

## ~/.claude/agents/api-designer.md

---
name: api-designer
description: RESTful API 设计专家，负责接口规范、版本策略和文档生成
model: claude-sonnet-4-6
tools:
  - Read
  - Write
  - Edit
  - Bash
---

你是一个资深的 API 设计专家，遵循 RESTful 最佳实践。

在设计 API 时你必须：
1. 严格遵守 HTTP 语义（GET 幂等，POST 创建，PUT 全量更新，PATCH 部分更新）
2. 使用统一的响应信封 { code, data, message, timestamp }
3. 版本号放在 URL 路径 /api/v1/
4. 错误码使用业务语义码（如 USER_NOT_FOUND），而非单纯 HTTP 状态码
5. 每个接口生成 OpenAPI 3.0 规范

当用户提供业务需求时，先输出接口列表，再生成代码。

调用自定义代理

# 两种调用方式

# 1. 在对话中明确声明
使用 api-designer 代理，为用户模块设计 CRUD 接口。

# 2. 通过 Agent 工具自动选择
# 当需求符合代理 description 时，Claude 会自动分配

Hooks 自动化

Hooks 是在 Claude 执行特定动作前后自动触发的 shell 命令，用于打造"格式化 → lint → 类型检查"的全自动质量守门员。

Hook 类型

类型触发时机典型用途
PreToolUse工具调用前验证、拦截危险操作
PostToolUse工具调用后自动格式化、lint、类型检查
Stop会话结束时最终构建验证、报告生成

配置 Hooks

在 ~/.claude/settings.json（全局）或 .claude/settings.json（项目级）中配置：

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "command": "prettier --write \"$FILE_PATH\"",
        "description": "自动格式化编辑的文件"
      },
      {
        "matcher": "Write|Edit",
        "command": "eslint --fix \"$FILE_PATH\"",
        "description": "自动修复 lint 问题"
      }
    ],
    "Stop": [
      {
        "command": "npm run build",
        "description": "会话结束前验证构建"
      }
    ]
  }
}

实用 Hook 配方

Java 项目：自动 checkstyle

{
  "matcher": "Write|Edit",
  "command": "mvn -q checkstyle:check -Dcheckstyle.failOnViolation=false"
}

Python 项目：black + ruff

{
  "matcher": "Write|Edit",
  "command": "black \"$FILE_PATH\" && ruff check --fix \"$FILE_PATH\""
}

防止大文件写入

多代理并行

Claude Code 最强大的能力之一：同时启动多个代理并行工作，把串行的小时级任务压缩到分钟级。

并行 vs 串行

场景串行（普通方式）并行（多代理）
代码审查质量审查 → 安全审查 → 性能审查（顺序等待）三个代理同时运行
大重构改前端 → 改后端 → 改测试（串行）三个模块独立代理并行
文档更新逐个文件更新多文件同时更新

启动并行代理

# 明确请求并行执行
同时启动三个代理并行工作：
1. code-reviewer 代理：审查 src/service/ 目录的代码质量
2. security-reviewer 代理：检查所有 API 接口的安全问题
3. typescript-reviewer 代理：检查类型定义完整性

三个代理同步进行，完成后汇总结果。

工作树隔离（Worktree）

并行任务需要操作同一仓库时，使用 git worktree 避免文件冲突：

# Claude 会自动创建隔离的工作树
我需要同时修复三个独立的 bug，请并行处理：
- Bug A：用户头像上传失败（isolation: worktree）
- Bug B：分页查询结果不稳定（isolation: worktree）
- Bug C：邮件通知未发送（isolation: worktree）

多视角分析模式

对重要架构决策，用多个角色代理各自独立分析，再综合：

对于"是否引入 Kafka 替换 Redis 消息队列"这个决策，
请同时启动以下代理从不同角度分析：
1. 架构师视角：系统复杂度和扩展性
2. 运维视角：运维成本和故障风险
3. 性能视角：吞吐量和延迟数据对比
最后综合三方意见给出建议。
ℹ️
并行代理的费用 并行代理同时消耗 token，成本是串行的 N 倍，但节省的时间远大于成本增加。对时间敏感的任务值得使用，日常简单任务串行即可。

MCP 扩展

MCP（Model Context Protocol）是 Claude 的标准化工具扩展协议，让 Claude 能直接连接数据库、API、浏览器、设计工具等外部系统。

常用 MCP 服务器

🗄️
数据库直连
PostgreSQL/MySQL MCP，让 Claude 直接查询表结构、运行 SQL、分析数据。
🌐
浏览器自动化
Playwright MCP，让 Claude 操作真实浏览器做 E2E 测试、UI 验证。
📁
Google Drive
读写 Google Docs/Sheets，把需求文档、数据直接引入 Claude 上下文。
📚
Context7 文档
实时拉取框架官方文档，解决 Claude 训练数据截止日期问题。
🔍
网络搜索
让 Claude 实时搜索最新的技术资料、库更新、错误解决方案。
🐛
Sentry / 监控
直接读取生产错误报告，Claude 可以基于真实错误数据分析问题。

配置 MCP 服务器

# 命令行快速添加
claude mcp add context7 -- npx -y @upstash/context7-mcp
claude mcp add playwright -- npx -y @playwright/mcp

# 查看已配置的 MCP
claude mcp list

在对话中使用 MCP

# 使用 Context7 查询最新文档
查询 React 19 中 useTransition 的最新用法（使用 Context7 文档服务）

# 使用数据库 MCP 直接分析
查询过去 30 天用户注册数据，分析注册漏斗，找出转化率最低的环节

# 使用浏览器 MCP 验证 UI
打开 http://localhost:3000/login，截图并检查登录表单是否正确渲染

分会话管理长任务

超大任务建议拆分为多个独立会话：

# 会话一：需求分析和设计
claude -p "分析需求文档，输出数据库设计和 API 接口列表，保存到 design.md"

# 会话二：后端实现（读 design.md）
claude -p "读取 design.md，实现后端接口，完成后更新 progress.md"

# 会话三：前端实现
claude -p "读取 design.md 和 progress.md，实现前端页面"

CLAUDE.md 写法

CLAUDE.md 是 Claude 的"工作手册"，写得越精准，Claude 的行为就越可预测、越贴合你的期望。

必须包含的内容

①

项目概述

是什么产品、核心功能、目标用户。帮助 Claude 理解决策背景。

## 项目概述
TeamFlow AI 是一个企业协同 SaaS 平台，覆盖项目管理、任务协作、
知识库、AI 助手等模块。主要用户是 B 端企业，5-500 人团队规模。
②

技术栈与命令

让 Claude 知道用什么工具、怎么运行。

## 技术栈
后端: Spring Boot 3.2 + JPA + Redis + MySQL 8
前端: Vue 3 + Vite + Pinia + TypeScript

## 常用命令
# 后端构建
mvn -q -DskipTests package
# 前端启动
cd frontend && npm run dev
③

代码规范

你团队的具体约定，比通用规范更重要。

## 代码规范
- API 响应统一格式: { code, data, message }
- 所有数据库操作走 Repository 层，禁止 Service 直接用 EntityManager
- 前端组件命名 PascalCase，文件名 kebab-case
- 禁止在 Controller 层写业务逻辑
④

禁止事项

明确告诉 Claude 什么不能做，避免意外改动。

## 禁止操作
- 不修改 deploy/ 目录下的 nginx.conf（运维管理）
- 不在代码中硬编码任何密钥或密码
- 不删除数据库字段（只能新增，废弃字段用 @Deprecated 标注）
- 不直接修改 main 分支，所有改动通过 PR

写法原则

原则差的写法好的写法
具体胜于抽象写高质量代码函数不超过 50 行，文件不超过 800 行
说明"为什么"不用 moment.js不用 moment.js（已弃用，改用 date-fns）
约束边界注意安全所有 SQL 使用参数化查询，禁止字符串拼接
定期更新一次写完不改每个 Sprint 结束复查，移除过时约定

掌握这些技巧，你与 Claude 的协作效率会提升 3-5 倍。

技巧一：给出参考样本

# 不说"按规范写"，而是给样本
参考 UserService.java 的风格，为 ProductService 实现
updateProduct() 方法。特别注意它的事务处理和错误处理方式。

技巧二：角色扮演约束

# 把 Claude 限定在特定角色
你现在是这个项目的资深后端工程师，对我提出的所有方案
先分析潜在风险，再给出实现建议。不要直接说"好的，我来实现"。

技巧三：分步骤确认

# 大任务分步完成，每步确认
我们要重构用户认证模块，分三步：
第一步：先列出所有需要改动的文件，不要动代码。
等我确认文件列表正确后，再进行第二步。

技巧四：明确输出格式

# 说清楚你要什么格式的输出
分析这段代码的性能问题，用以下格式输出：
**问题：** [问题描述]
**影响：** [严重程度和影响范围]
**解决方案：** [代码级别的解决方案]
**预期收益：** [优化后的效果]

技巧五：提供失败案例

# 告诉 Claude 什么方向走不通
我们已经尝试过用 Redis 分布式锁解决这个并发问题，
但在高并发下锁超时导致数据不一致。
请给出不依赖 Redis 锁的替代方案。

技巧六：渐进式修改

# 大改动分小批次
先只修改 UserController.java 中的 createUser 方法，
其他方法不要动。改完后我看看效果再决定是否继续。

常见坑与避坑

这些是新手最容易踩的坑，提前了解可以帮你少走很多弯路。

坑一：太大的一次性任务

❌
错误做法 "帮我把整个项目从 Vue 2 迁移到 Vue 3"——这会导致 Claude 修改数十个文件，中途上下文耗尽，最终一团糟。
✅
正确做法 按模块拆分：先迁移 src/components/Button.vue，验证通过后再继续下一个组件。每次只处理一个可验证的单元。

坑二：不验证 AI 输出就合并

Claude 有时会：

  • 生成看起来正确但逻辑有缺陷的代码
  • 引入不需要的依赖
  • 修改你没让它动的文件
  • 解决了表面问题但留下根本原因

避坑：每次改动后运行测试，用 git diff 仔细检查实际变更范围。

坑三：忽视 CLAUDE.md 维护

项目架构变化、技术选型更新、规范调整——如果不同步更新 CLAUDE.md，Claude 会基于过时信息给出错误建议。

避坑：把更新 CLAUDE.md 列入 Sprint 完成清单。每次重大决策后立即更新。

坑四：在上下文快满时做大决策

上下文超过 80% 时，Claude 的"注意力"已经分散。这时让它做架构决策或大规模修改，错误率会显著上升。

避坑：定期用 /compact 压缩，或开新会话处理新任务。

坑五：用 AI 代替思考

Claude 是执行者，不是决策者。它会根据你的指令行动，但不理解你业务的真正优先级和长期影响。

⚠️
Claude Code 的边界 架构决策、技术选型、产品优先级——这些是你的责任。Claude 负责执行，你负责方向。不要因为 AI 能做很多事，就把思考权也转移出去。
