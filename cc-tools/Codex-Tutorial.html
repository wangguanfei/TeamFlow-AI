入门

什么是 Codex

Codex 是 OpenAI 面向软件开发的编码代理。它可以在本地项目、IDE、桌面应用和云端环境中读代码、改文件、运行命令、调用插件与 MCP 工具，并把开发过程推进到可验证的结果。

核心心智模型

01
不是补全，是执行代理
Codex 可以直接进入仓库读写文件、运行测试、查看 diff，并根据结果继续修正。
02
多入口共享配置
CLI、IDE Extension 和 Codex App 共享 `~/.codex/config.toml`、MCP 和技能等配置。
03
能力由层组成
AGENTS.md 定规则，Skills 定流程，Plugins 分发能力，MCP 连接外部系统，Hooks 做生命周期自动化。
04
验证是完成标准
让 Codex 跑测试、打开浏览器、看日志、检查健康接口，而不是只输出“已完成”。

Codex 与 Claude Code 的关键差异

维度Claude Code 常见做法Codex 推荐做法
项目规则CLAUDE.mdAGENTS.md，支持全局、仓库和子目录层级
配置入口~/.claude/settings.json~/.codex/config.toml 与项目 .codex/config.toml
扩展分发技能、命令、代理、MCPSkills 是工作流，Plugins 是安装分发单元，可带 MCP、apps、hooks
本地安全工具 allowlist 和 hooks审批策略、sandbox、permissions profile、rules、hooks 一起约束
桌面能力终端与 IDE 为主Codex App 支持多线程、工作树、插件、自动化、浏览器和桌面应用操作
i
使用原则 把 Codex 当成一个有工具权限的工程师。你负责目标、边界和验收，Codex 负责读代码、修改、执行、验证和反馈。

入口与安装配置

Codex 有 App、CLI、IDE Extension 和 Cloud 几个常用入口。选入口不是偏好问题，而是任务环境问题。

入口选择

入口适合场景典型动作
Codex App日常项目开发、多线程、工作树、插件、自动化、浏览器验证打开项目目录，选择 Local / Worktree / Cloud
Codex CLI终端开发、SSH、脚本化、非交互任务codex、codex exec、codex mcp
IDE Extension围绕当前文件、选区、诊断快速协作带入 IDE context 让 Codex 修改当前文件
Codex Cloud后台任务、CI 修复、PR 审查、长任务从 App、GitHub 评论或 CLI 提交云任务

首次配置建议

1

确认认证与环境

登录后先确认 CLI 和 App 都能看到同一个项目环境。

codex login
codex doctor --summary
codex app /your/project
2

设置安全默认值

新手优先使用 workspace-write 和 on-request。需要更宽权限时按命令临时审批。

# ~/.codex/config.toml
approval_policy = "on-request"
sandbox_mode = "workspace-write"
3

接入项目规则

用 /init 生成 AGENTS.md，再手动补充真实命令和禁区。

codex
# 在 Codex 中执行
/init
!
不要一上来开最大权限 danger-full-access 只适合外部已经隔离的环境。日常项目开发用 workspace-write，再对必要命令做明确审批。

子目录规则

Codex 会从项目根向当前目录加载指导，越靠近当前目录越具体。大型项目建议按模块放置局部 AGENTS.md。

repo/
├── AGENTS.md              # 全局规范
├── backend/
│   └── AGENTS.md          # 后端测试、数据库、事务规则
├── frontend/
│   └── AGENTS.md          # 组件、样式、浏览器验证规则
└── deploy/
    └── AGENTS.md          # 部署和回滚规则

输出一份项目开发入口说明：常用命令、关键目录、测试方式、风险区域。

i

使用 /plan

/plan 为 AI 知识库新增“按项目检索范围”能力。
请先阅读相关代码，输出：
1. 涉及文件
2. 数据流变化
3. 后端和前端改动步骤
4. 测试计划
5. 风险和回滚点
先不要改代码。

使用 /goal

/goal 完成知识库检索范围改造：
- 只改 AI 模块相关文件
- 改完跑后端相关测试和前端构建
- 给出验证命令和结果
- 未经我确认不要部署
i
目标模式适合长任务 当任务需要多轮读代码、实现、测试、修复时，用 /goal 保持 Codex 的终点稳定。

高质量实现提示词

按刚才的计划实现第一阶段：
范围：

让 Codex 读现有模式

不要让 Codex凭空设计。让它先找相邻实现，照项目风格改。

先找项目里已有的分页查询、错误处理和权限校验写法。
实现新接口时必须复用现有响应结构和异常风格。

控制改动粒度

A

确定 DTO、接口参数和兼容默认值，再写实现。
B
再业务逻辑
服务层、查询、事务和错误处理小步落地。
C
再前端接入
类型、API、状态、页面交互逐层验证。
D
最后收尾
删临时代码、补测试、看 diff、总结风险。
!
警惕“顺手优化” 如果 Codex开始重构无关文件，立刻收窄范围。AI 改动越宽，验证成本越高。

验证矩阵

改动类型最低验证更可靠验证

3. 打开页面做浏览器验证
4. 如果失败，先分析失败原因，再决定是否修代码

浏览器验证提示词

使用 Browser 打开本地页面：
- 桌面宽度 1440
- 移动宽度 390

本地审查

/diff
/review 请审查当前工作区改动：
1. 优先找 bug、回归、安全风险和缺失测试
2. 标出文件和行号
3. 不要列无关风格建议
4. 如果没有问题，说清楚剩余风险

提交原则

  • 不要让 Codex 使用 git add .，要求它按文件确认 stage。
  • 脏工作区里只 stage 本次任务文件。
  • 提交信息写清楚类型和范围，例如 feat(cc-tools): 新增 Codex 能力页。
  • 提交前让 Codex 复述验证命令和结果。

GitHub 协作

使用 GitHub 插件检查当前 PR：
- 未解决 review thread
- 失败的 checks
- 最近一次 CI 日志关键错误
先汇总，再修复可操作问题。
!
审查不等于测试 代码审查能发现风险，但不能证明行为正确。提交前仍然需要真实验证结果。

Skills 与 Plugins

Skills 定义“怎么做”，Plugins 负责“怎么分发和接入外部能力”。不要把所有东西都塞进 AGENTS.md。

什么时候用 Skill

  • 同一类任务反复出现，例如发布检查、代码审查、报告生成。
  • 流程需要参考资料、脚本或固定输出格式。
  • 希望 Codex 能根据描述自动选择工作流。
# .agents/skills/release-check/SKILL.md
---
name: release-check
description: 发布前检查技能。用于构建、测试、配置、健康检查和回滚核对。
---

1. 读取 AGENTS.md 和部署文档。
2. 列出发布风险和必须验证命令。
3. 运行允许范围内的检查。
4. 输出发布结论和剩余风险。

什么时候用 Plugin

当你想把多个 skills、MCP、app connector、hooks、资产打包给自己或团队复用时，用 Plugin。

需求推荐
只是一段项目规则AGENTS.md
一个可复用工作流Skill
工作流 + 外部工具 + 团队分发Plugin
连接外部系统MCP 或 App Connector，可由 Plugin 打包

常用插件场景

GH
GitHub
PR、issue、CI、review thread、发布协作。
BR
Browser / Chrome
本地页面验证、登录态浏览器、截图和交互。

Figma

MCP 外部系统

MCP 是 Codex 连接外部工具和上下文的标准方式。它适合“信息不在仓库里”的任务。

配置方式

# CLI 添加 stdio MCP
codex mcp add context7 -- npx -y @upstash/context7-mcp

# 查看 MCP 状态
codex mcp list

# 在 TUI / App 中查看当前会话
/mcp

config.toml 示例

# ~/.codex/config.toml
[mcp_servers.context7]
command = "npx"
args = ["-y", "@upstash/context7-mcp"]

[mcp_servers.figma]
url = "https://mcp.figma.com/mcp"
bearer_token_env_var = "FIGMA_OAUTH_TOKEN"

常用 MCP 选择

MCP解决什么问题典型提示词
Context7最新框架文档用 Context7 查询 Next.js 当前路由写法
Playwright真实浏览器验证打开登录页，检查移动端布局
MySQL / PostgreSQL表结构和查询分析查看订单表索引，解释慢查询
Redis缓存和队列状态检查登录 session key 的 TTL
Figma设计稿和组件读取设计 token，生成页面实现
Sentry线上异常读取最近错误，定位回归点
i
MCP 最好和 Skill 配套 Skill 说明流程，MCP 提供工具。比如“前端验收”技能可以固定要求使用 Browser 或 Playwright MCP。
扩展能力

Hooks 自动化

Hooks 是 Codex 生命周期里的脚本扩展点。它适合机械、确定、可审计的检查，不适合塞复杂业务判断。

常用事件

事件触发时机适合用途
PreToolUse工具调用前拦截危险命令、校验路径、检查策略
PermissionRequest请求权限时判断是否访问生产、是否写出工作区
PostToolUse工具调用后格式化、lint、扫描命令输出
UserPromptSubmit用户提交提示后扫描密钥、补充上下文、审计
Stop任务停止时最终检查、报告、提醒未验证项

hooks.json 示例

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "/usr/bin/python3 .codex/hooks/pre_tool_use_policy.py",
            "statusMessage": "Checking command policy"
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "apply_patch|Edit|Write",
        "hooks": [
          { "type": "command", "command": "npm run lint --if-present", "timeout": 60 }
        ]
      }
    ]
  }
}

Hook 设计原则

  • 只做确定性检查，不让 hook 生成大段不可控改动。
  • 脚本输出要短，失败原因要清楚。
  • 高风险 hook 放用户级或受信项目级配置，不随便接受陌生仓库 hooks。
  • 新 hook 需要在 /hooks 里 review 和 trust。
扩展能力

自动化与工作树

Automations 适合周期性任务，Worktree 适合隔离并行改动。两者配合可以把例行检查从主工作区移走。

三种自动化

类型特点适用场景
Thread Automation回到同一个线程等待部署、轮询 CI、继续同一调查
Standalone Automation每次独立运行日报、巡检、依赖提醒、PR 汇总
Project Worktree Automation在独立工作树运行可能产生改动的定时任务

自动化提示词模板

每天上午 9 点检查这个项目：
1. 是否有失败的 GitHub Actions
2. 是否有未处理 PR review
3. 是否有依赖安全告警
只报告需要我处理的事项。
如果没有发现，自动归档本次运行。

Worktree 使用建议

  • 探索性重构、依赖升级、并行 bug 修复优先用 Worktree。
  • 本地主工作区有未提交改动时，不要让后台任务直接在 Local 模式改文件。
  • 任务完成后再把 diff 合并回主工作区或开 PR。
!
自动化也受权限影响 自动化无人值守运行，默认沙箱和审批策略要更保守。需要写文件或联网时，先手动跑通一次。
最佳实践

安全与权限

Codex 能执行真实命令，所以安全设计要前置。默认约束要保守，放权要有理由和范围。

权限分层

FS
文件系统
read、write、deny 控制路径访问。敏感目录和 .env 应显式保护。
NET
网络
默认可关闭或按域名允许。生产、内网、Docker socket 要谨慎。
APP
应用连接
插件和 connectors 可能访问私有数据，按任务最小授权。
CMD
命令审批
安装依赖、部署、删除、越权写入、联网失败重试都应明确审批。

常见安全提示词

开始前先确认：
1. 本次会修改哪些目录
2. 是否需要网络
3. 是否需要访问密钥或生产环境
4. 哪些命令需要我审批
如果可以在当前 workspace-write 范围内完成，就不要请求额外权限。

密钥与日志

  • 不要把 API key、cookie、证书、数据库密码贴进提示词。
  • 需要用密钥时，优先让系统从环境变量读取。
  • 让 Codex 在提交前扫描 diff，确认没有密钥、token、客户数据。
  • 不要让记忆或文档沉淀短期凭证。

迁移提示词

请检查当前 Claude Code 配置，并按 Codex 层级给出迁移计划：
1. CLAUDE.md → AGENTS.md
2. settings.json → config.toml
3. slash commands → skills
4. MCP → codex mcp/config.toml
5. hooks → hooks.json
6. 哪些内容不应迁移
先输出计划，不要写文件。

迁移后验证

  • 新开 Codex 线程，让它列出加载到的 AGENTS.md 来源。
  • 用 /mcp 验证外部工具是否可用。
  • 用 /hooks review 和 trust 迁移后的 hooks。
  • 跑一个小任务验证技能是否被正确触发。
  • 不要把 Claude 的危险 allowlist 原样迁移成 Codex 宽权限。
